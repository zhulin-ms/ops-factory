# 协同开发规范说明

为降低跨团队协作成本、减少返工，并避免隐性约束在人员流动或 AI 新 session 中丢失，仓库已补充一套面向研发协同的文档结构。目标不是增加流程负担，而是把原来依赖口头沟通、代码习惯和历史讨论的规则，沉淀为可查、可执行、可复用的约束。

这套规范主要解决三个问题：

1. 新同事或新团队接手时，不需要依赖“问人”才能理解系统边界。
2. 多团队并行开发时，减少对架构、接口、UI 交互方式的误改。
3. 使用 AI 辅助开发时，让模型有稳定的仓库内规则来源，而不是每次重新猜测。

## 文档分工

- `README.md`：项目介绍、快速启动、文档入口。
- `AGENTS.md`：所有贡献者和 AI 都应优先遵守的短约束。
- `docs/architecture/*`：系统边界、API 契约、进程管理等架构规则。
- `docs/development/*`：开发规范、UI 约束、测试要求、评审清单。
- `docs/operations/*`：运维、排障、故障复盘和最佳实践。

## 团队实践要求

1. 开始开发前，至少阅读 `AGENTS.md`、`docs/architecture/overview.md`，以及与本次改动相关的 development 文档。
2. 如果改动影响 API、SSE、鉴权头、配置项、运行时目录、UI 交互骨架，必须在同一个 PR 中同步更新对应文档。
3. 新增配置项时，必须同时更新代码实现、对应的配置示例文件和相关文档。
4. 不允许绕过既定架构边界。前端不得直连 provider 或 agent runtime，浏览器侧访问必须继续通过 gateway。
5. UI 改动必须遵守现有交互模型，包括路由结构、侧边栏、右侧面板、错误提示和 i18n 方式。若要突破这些约束，需在 PR 中明确说明原因和影响。
6. 测试要和改动范围匹配：前端逻辑补 Vitest，Java 行为补 JUnit，跨服务行为补 integration test，主用户流程改动补 Playwright。
7. 提交和评审 PR 时，统一参考 `docs/development/review-checklist.md`，不要只检查代码是否“能跑”。
8. 当某个规则已经成为团队共识时，应先写入 `AGENTS.md` 或 `docs/`，再要求其他团队或 AI 遵守，不再依赖口头传达。

## 建议执行方式

- 将文档更新纳入代码评审检查项。
- 将这些文档纳入新成员 onboarding 必读材料。
- 使用 AI 辅助开发时，明确要求优先遵守仓库内 `AGENTS.md` 和 `docs/`。
- 当某类返工或误解重复出现时，及时补充显式规则，而不是继续靠经验传递。

## 结论

这套规范的核心不是“增加文档”，而是把跨团队协作中的关键约束前置、显式化、仓库化。后续所有团队成员和 AI 助手都应以这些文档为统一参考，确保项目在多人、多团队、长周期演进中仍能保持一致性和可维护性。
